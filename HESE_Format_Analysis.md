# HESE Format Analysis

이 문서는 `script_out/*.hese` 파일을 기존 `tools` 코드와 샘플 파일 관찰 결과를
바탕으로 정리한 분석 메모다. 현재 목적은 포맷을 완전히 디컴파일하는 것이 아니라,
로컬라이징 작업에서 이 파일이 무엇이고 어느 부분을 조심해야 하는지 파악하는 데 있다.

## 요약

`script_out` 폴더의 `.hese` 파일은 `script.heslnk`에서 추출된 개별 시나리오
스크립트다. `.hese` 자체는 대사 본문 테이블이 아니라, HuneX/UNAS 계열 엔진에서
실행하는 VM 바이트코드로 보인다.

대사 본문은 보통 `script_dialog_*.hdlg` 쪽에 있고, `.hese`는 배경, 음악, 효과음,
캐릭터, 대사 ID, 분기, 연출 호출 등을 실행 순서대로 담는 역할에 가깝다.

## 관련 파일 관계

### `script.heslnk`

`script.heslnk`는 여러 `.hese` 스크립트를 담는 컨테이너다.
기존 `tools/heslnk_codec.py` 기준 구조는 다음과 같다.

- `HESL` 매직을 가진 아카이브
- 엔트리 테이블에 파일명 CRC32, HLZS 블록 오프셋, 압축 해제 크기를 저장
- 이름 테이블은 NUL 종료 UTF-8 문자열
- 각 데이터 블록은 `HLZS` 압축된 `.hese`
- 데이터 블록은 0x20 정렬

즉 `script_out/*.hese`는 `script.heslnk` 안의 `HLZS` 블록을 LZSS로 압축 해제한
결과물이다.

### `script_dialog_*.hdlg`

`tools/hdlg_codec.py`는 `HLZS`로 감싼 `HDGL` 대사 테이블을 다룬다.
이쪽은 NUL 종료 UTF-8 문자열 목록을 가진 텍스트 테이블이며, 실제 번역 대상 대사는
대체로 여기에 있다.

`.hese` 안에는 대사 본문이 직접 들어 있다기보다 `sc_hol01_0010_ch02_057` 같은
대사 ID 또는 이벤트 ID가 들어 있고, 런타임에서 이 ID를 `HDGL` 대사 테이블과 연결해
표시하는 구조로 추정된다.

## 관찰된 HESE 헤더

전체 597개 `.hese` 파일을 확인했을 때 공통적으로 다음 패턴을 가진다.

| 오프셋 | 크기 | 값/의미 |
| --- | ---: | --- |
| `0x00` | 4 | 매직 `HESE` |
| `0x04` | u32 | 버전 `0x00010200` |
| `0x08` | u32 | 코드 시작 오프셋. 전 파일에서 `0x40` |
| `0x0c` | u32 | 관찰값 `0` |
| `0x10` | u32 | 해당 스크립트의 대사 슬롯 수로 추정 |
| `0x14` | u32 | 관찰값 `0` |
| `0x18` | u32 | 관찰값 `0` |
| `0x1c` | u32 | 관찰값 `0` |
| `0x20` | u32 | 전체 대사 슬롯 공간에서 이 스크립트가 시작하는 base offset |
| `0x24`-`0x3f` | bytes | 현재 샘플에서는 대부분 0 padding |
| `0x40` | bytes | VM 바이트코드 시작 |

예시:

| 파일 | `0x10` count | `0x20` base |
| --- | ---: | ---: |
| `sc_end01_ch01_0010.hese` | 110 | `0x0000` |
| `sc_end01_ch02_0010.hese` | 128 | `0x0080` |
| `sc_map0101a_pl01_0010.hese` | 9 | `0x2200` |
| `sc_hol01_0010.hese` | 596 | `0x1440` |

`0x20` base는 파일명 정렬 순서 기준으로 이전 파일들의 `0x10` count를 0x20 단위로
올림 정렬하며 누적한 값과 일치했다. 전체 누적 최종값은 `0x6240`이고,
`script.heslnk` 헤더의 `unknown2` 값도 `0x6240`이었다. 따라서 이 값은 전체 대사
슬롯 또는 대사 참조 테이블 크기와 관련된 값으로 보는 것이 자연스럽다.

## 바이트코드 본문

`0x40` 이후는 VM 명령어와 피연산자가 섞인 바이트코드다.

관찰상 `0x40`부터 `0x1ce03`까지는 모든 `.hese` 파일에서 동일한 공통 코드 영역이다.
이 영역은 카메라, 배경, 캐릭터, 사운드, 선택지, 대사 표시 등 엔진의 공통 헬퍼 함수
또는 공통 스크립트 루틴으로 보인다.

`0x1ce04` 이후부터 각 파일별 실제 시나리오 코드가 달라진다. 여기에는 다음과 같은
문자열이 자주 등장한다.

- 배경: `BG000_00`, `bg026_02`, `bg071_01`
- BGM: `bgm067`, `bgm098`, `bgm103`
- 효과음: `se_110_0012`, `se_110_0070`
- 캐릭터/표정/포즈: `ch01_72M_02`, `ch06_04M_01`
- 스크립트 이동: `sc_mor0201_0010`, `sc_map`
- 대사/이벤트 ID: `sc_hol01_0010_ch02_057`, `SYSTEM_0114_0770`

문자열 앞에는 `a0 2d` 계열 패턴이 자주 붙고, 정수/float 상수에는 `a0 23 00 90`,
`a0 23 00 88` 같은 패턴이 반복된다. 정확한 opcode 의미는 아직 확정하지 않았지만,
값 push, 함수 호출, 객체/파라미터 구성 명령으로 추정된다.

## 대사 참조 패턴

일부 일반 시나리오 파일에서는 다음과 비슷한 패턴이 반복된다.

```text
a0 1b 00 90 01 00 00 00 [local_index] [dialog_id]
```

예를 들어 `sc_hol01_0010.hese`는 헤더 count가 596이고, 해당 패턴도 대사 순서에
맞춰 대량으로 등장한다. `local_index`는 파일 내부 대사 순번, `dialog_id`는
`HDGL` 대사 테이블 쪽 ID 또는 전역 대사 번호로 추정된다.

단, `sc_nig_*` 계열처럼 같은 방식으로 잡히지 않는 파일도 있다. 따라서 이 패턴만으로
모든 대사 참조를 안정적으로 추출할 수 있다고 보기는 어렵다.

## 로컬라이징 관점의 의미

번역 본문 수정은 우선 `script_dialog_*.hdlg`를 대상으로 하는 것이 맞다.
기존 `hdlg_to_xlsx.py`, `xlsx_to_hdlg.py`, `verify_hdlg_roundtrip.py` 흐름은 이 목적에
맞게 구성되어 있다.

`.hese`는 대사 문장을 직접 번역하는 파일이라기보다 다음 항목을 제어한다.

- 어느 대사 ID를 호출하는지
- 배경/음악/효과음/캐릭터 리소스를 언제 호출하는지
- 선택지, 분기, 이동 대상이 무엇인지
- 화면 전환, 페이드, 카메라, UI 연출이 어떻게 실행되는지

따라서 `.hese`를 수정해야 하는 경우는 보통 다음과 같은 작업이다.

- 대사 호출 순서 또는 대사 ID 연결 수정
- 분기/선택지 흐름 수정
- 리소스 이름 변경에 따른 참조 수정
- 특정 연출 호출 제거 또는 교체

단순 번역문 교체만이면 `.hese`를 직접 건드릴 가능성은 낮다.

## 주의점

`.hese`는 일반 텍스트 포맷이 아니며, 문자열 길이를 바꾸면 주변 오프셋 또는 명령어
피연산자가 깨질 수 있다. 문자열을 직접 hex 편집하는 방식은 위험하다.

현재 repo의 `heslnk_tool.py`는 `.hese`를 추출하고 다시 `script.heslnk`로 묶는 데는
사용할 수 있지만, `.hese` 내부 VM 바이트코드를 의미 단위로 파싱하거나 재빌드하는
도구는 아직 없다.

안전한 작업 순서는 다음과 같다.

1. 대사 본문은 `HDLG` 변환 도구로 수정한다.
2. `.hese`는 가능하면 원본 바이트를 유지한다.
3. `.hese` 수정이 필요하면 먼저 참조 문자열과 주변 opcode를 분석한다.
4. 수정 후에는 `heslnk_tool.py repack` 및 `verify`로 컨테이너 라운드트립을 확인한다.
5. 실제 게임에서 해당 장면을 실행해 런타임 오류와 대사 표시를 확인한다.

## 현재 결론

`script_out/*.hese`는 `script.heslnk`에서 추출된 HESE VM 스크립트다.
대사 본문 데이터라기보다 장면 진행과 연출, 리소스 호출, 대사 ID 호출을 담당한다.

번역 작업의 1차 대상은 `script_dialog_*.hdlg`이며, `.hese`는 대사 연결이나 연출 흐름을
수정해야 할 때 별도 분석 후 다루는 것이 좋다.
